Abuse Case

Introduction

Often when the security level of an application is mentioned in requirements, the following expressions are met:

  • The application must be secure.
  • The application must defend against all attacks targeting this category of application.
  • The application must defend against attacks from the OWASP TOP 10
  • ...

These security requirements are too generic, and thus useless for a development team...

In order to build a secure application, from a pragmatic point of view, it is important to identify the attacks which the application must defend against, according to its business and technical context. Abuse cases are a frequently recommended threat modeling tool, and reviewing the threat modeling cheat sheet may be helpful.

Objective

The objective of this cheat sheet is to provide an explanation of what an Abuse Case is, how abuse cases can be important when considering the security of an application, and finally to provide a proposal for a pragmatic approach to building a list of abuse cases and tracking them for every feature planned for implementation as part of an application. The cheat sheet may be used for this purpose regardless of the project methodology used (waterfall or agile).

The main objective is to provide a pragmatic approach in order to allow a company or a project team to start building and handling the list of abuse cases and then customize the elements proposed to its context/culture in order to, finally, build its own method. This cheat sheet can be seen as a getting-started tutorial.

Context & approach

Why clearly identify the attacks

Clearly identifying the attacks against which the application must defend is essential in order to enable the following steps in a project or sprint:

  • Evaluate the business risk for each of the identified attacks in order to perform a selection according to the business risk and the project/sprint budget.
  • Derive security requirements and add them into the project specification or sprint's user stories and acceptance criteria.
  • Estimate the overhead of provision in the initial project/sprint charge that will be necessary to implement the countermeasures.
  • About countermeasures: Allow the project team to define them, and to determine in which location (network, infrastructure, code...) they should be located.

Notion of Abuse Cases

You can think of Abuse cases in two ways. The first is to discover attacks (answer the question "what can go wrong"), and the second is to help record those attacks (informally, this includes threats, issues, risks) in a form that may be less intimidating to developers.

An Abuse Case can be defined as:

A way to use a feature that was not expected by the implementer, allowing an attacker to influence the feature or outcome of use of the feature based on the attacker action (or input).

Synopsis defines an Abuse Case like this:

Misuse and abuse cases describe how users misuse or exploit the weaknesses of controls in software features to attack an application. This can lead to tangible business impact when a direct attack against business functionalities, which may bring in revenue or provide positive user experience, are attacked. Abuse cases can also be an effective way to drive security requirements that lead to proper protection of these critical business use cases.

How to define the list of Abuse Cases

There are many different ways to define the list of abuse cases for a feature (that can be mapped to a user story in agile projects). Threat Modeling is a set of techniques for anticipating what can go wrong, and ensuring we do something about each identified possible scenario. Taking each item on the list of "what are we going to do about it" and writing an abuse case may help your engineering teams process the output.

Practical Approach to Creating Abuse Cases

Here's a step-by-step approach to creating and managing abuse cases:

  1. Identify Features: List all features and functionality in your application that could be potential targets for abuse.
  2. Brainstorm Attack Scenarios: For each feature, consider:
    • How could an attacker misuse this feature?
    • What unintended consequences could occur?
    • What business logic could be bypassed?
    • What data could be exposed or manipulated?
  3. Document Abuse Cases: For each identified scenario, create a structured abuse case that includes:
    • Title/ID: Unique identifier for the abuse case
    • Description: Clear explanation of the potential abuse
    • Impact: Business and technical impact if exploited
    • Risk Level: High/Medium/Low based on impact and likelihood
    • Mitigation: Required controls or countermeasures

Example Abuse Cases

Example 1: User Registration

Feature: User registration with email verification

Abuse Case: Account Enumeration

  • Scenario: Attacker attempts to register with known email addresses to determine which ones are already registered
  • Impact: Privacy violation, potential for targeted attacks
  • Mitigation: Use consistent response times and messages for both existing and non-existing accounts
Example 2: Password Reset

Feature: Password reset functionality

Abuse Case: Token Reuse

  • Scenario: Attacker captures a password reset token and attempts to reuse it after it's been used
  • Impact: Account takeover
  • Mitigation: Implement one-time use tokens with short expiration times

Integration with Development Lifecycle

To effectively implement abuse cases in your development process:

  • Requirements Phase: Include abuse cases in feature specifications
  • Design Phase: Review abuse cases during architecture and design reviews
  • Implementation Phase: Use abuse cases to guide security testing
  • Testing Phase: Create test cases based on abuse cases
  • Maintenance Phase: Regularly review and update abuse cases as new threats emerge

Best Practices

  • Keep abuse cases simple and focused on specific scenarios
  • Use clear, non-technical language when possible
  • Regularly review and update abuse cases
  • Include both technical and business impact in assessments
  • Link abuse cases to specific security requirements
  • Maintain a centralized repository of abuse cases
  • Involve all stakeholders in abuse case creation and review

Common Pitfalls to Avoid

  • Creating too many or too vague abuse cases
  • Focusing only on technical vulnerabilities
  • Neglecting business logic abuse scenarios
  • Not updating abuse cases as the application evolves
  • Failing to prioritize abuse cases based on risk
  • Not involving all relevant stakeholders

Tools and Resources

  • OWASP Threat Modeling Cheat Sheet
  • OWASP Top 10 for reference on common vulnerabilities
  • STRIDE methodology for threat modeling
  • Attack trees for visualizing attack scenarios
  • Security requirement templates

Conclusion

Abuse cases are a powerful tool for identifying and mitigating security risks in your application. By systematically identifying potential abuse scenarios and implementing appropriate controls, you can significantly improve your application's security posture. Remember that abuse cases should be living documents that evolve with your application and the threat landscape.

Note: This cheat sheet provides a starting point for implementing abuse cases in your development process. Customize the approach based on your organization's needs, risk tolerance, and development methodology.
